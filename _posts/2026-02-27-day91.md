## Git/GitHub Key Functionalities (Based on My Current Understanding)

- Git helps me track code history, compare changes, and prepare clean commits.
- GitHub is where I store and collaborate on Git repositories online (sharing, pull requests, code review, backup).

### 1) Viewing Commit History
- Command: `git log --since='Apr 2 2024' --until='Apr 11 2024'`
- Meaning: show commits created between those two dates.

### 2) Comparing Changes
- Command: `git diff`
- Correct meaning: shows unstaged changes (working directory vs staging area), not strictly "difference from the last commit."
- If I want to compare against the last commit, I can use: `git diff HEAD`

### 3) Understanding Diff Header
- Example: `@@ -3,15 +3,33 @@`
- Correct meaning:
  - `-3,15` = in the old version, this hunk starts at line 3 and covers 15 lines
  - `+3,33` = in the new version, this hunk starts at line 3 and covers 33 lines
- It does **not** mean "line 3 is deleted."
- Actual deletions are lines prefixed with `-`, additions are lines prefixed with `+`.

### 4) Staging and Reviewing Staged Changes
- Command: `git add file.md`
- Meaning: place the current version of `file.md` into the staging area for the next commit.
- Command: `git diff --staged file.md`
- Meaning: compare staged content of `file.md` against the last commit (what will be committed).

### Why `-a` Sometimes Works in Git

Short answer: for `git add`, the common flag is `-A` (capital A), not `-a`.

- `git add -A` = stage **all** changes (new, modified, deleted).
- `git add .` = stage changes under the current directory.
- `git commit -a` (lowercase `-a`) is different: it auto-stages **tracked** modified/deleted files, but **not** new untracked files.

`git add` / `git commit` change your local repo only. Until `git push origin main`, GitHub online does not change.

## Before commit (you already ran `git add file.md`)

- Unstage only: `git restore --staged file.md`
- Discard local edits and go back to your current local commit (`HEAD`): `git restore file.md`
- Make local file exactly match GitHub `main`:
  1. `git fetch origin`
  2. `git restore --source origin/main --staged --worktree file.md`

## After commit

- If commit is **not pushed**: you can rewrite local history, e.g. `git reset --hard HEAD~1` (destructive).
- If commit is **already pushed** (or shared): use `git revert <commit>` (safe; creates a new commit that undoes it).

## What `git checkout HEAD~1 -- report.md` means

- Think of it as: "replace my current `report.md` with the older version from 1 commit ago."
- `HEAD~1` means "the commit right before my current commit."
- Only `report.md` is changed (not all files).
- After this, commit if you want to keep that rollback.
- Same command in modern style: `git restore --source=HEAD~1 --staged --worktree report.md`.

## What `git revert` means

- “Undo by adding a new commit,” not by deleting history.
    - `git revert <commit>` undoes the changes introduced by that specific commit (which may include multiple files).
    - Git then creates a new commit that records this undo.
    - Original commit stays in history; it is not deleted.

## Case: `git add` used, but not `git commit` yet

You are in the **staging** phase. Nothing is in commit history yet.

### Unstage files
- Unstage a single file: `git restore --staged summary_statistics.csv`
- Unstage all files: `git restore --staged .`

### Typical workflow
1. Edit the file: `summary_statistics.csv`
2. Stage it: `git add summary_statistics.csv`
3. (Optional) Unstage one file: `git restore --staged summary_statistics.csv`
4. (Optional) Unstage all files: `git restore --staged .`
5. Stage again after more edits: `git add summary_statistics.csv`
6. Commit: `git commit -m "Adding age summary statistics"`

### Key point
- `git restore --staged` only affects the staging area.
- Your file content is still there unless you also run a discard command like `git restore summary_statistics.csv`.


## Git Undo/Staging Summary

| Command | Result |
|---|---|
| `git revert HEAD` | Revert the latest commit by creating a **new commit** that undoes it (all files changed in that commit). |
| `git revert HEAD --no-edit` | Same as above, but skip opening the commit message editor. |
| `git revert HEAD -n` | Apply the revert changes to your working tree/index **without creating a commit yet**. |
| `git checkout HEAD~1 -- report.md` | Restore only `report.md` from the previous commit into your working tree (and stage it). |
| `git restore --staged report.md` | Unstage one file (remove it from staging area, keep local edits). |
| `git restore --staged .` | Unstage all staged files. |

### Notes
- `git revert` is safe for shared history because it does not delete old commits.
- `git restore --staged` needs a path (for all files, use `.`).

## 1) Branches: what + why

### What is a branch?
- A **branch** = an individual version of a repo.
- Git uses branches to systematically track multiple versions of files.
- In each branch:
  - some files may be the same
  - others may be different
  - some may not exist at all :contentReference[oaicite:3]{index=3}

### Why use branches?
Common reasons (from the slides):
- Protect the **live system**: default branch is typically `main` (the “works as expected” version). :contentReference[oaicite:4]{index=4}
- Develop features safely: issues during development/testing won’t affect the live system. :contentReference[oaicite:5]{index=5}
- Multiple developers can work simultaneously.
- Compare repo state between branches.
- Combine contents to push new features to the live system.
- Each branch should have a specific purpose. :contentReference[oaicite:6]{index=6}

### Mental model (what the diagrams show)
- You can **branch off** from `main` into a feature branch (example shown: `ai-assistant`). :contentReference[oaicite:7]{index=7}
- Later, you **merge back** into `main`. :contentReference[oaicite:8]{index=8}
- For bugs, you might create a separate `bug-fix` branch while feature work continues. :contentReference[oaicite:9]{index=9}

## Git Notes: Clean Learning Guide

This note summarizes practical Git and GitHub workflows from beginner to intermediate level.

## 1) Core idea: local Git vs GitHub remote

- `git add` and `git commit` only change your **local** repository.
- GitHub is unchanged until you run a push command such as `git push origin main`.

Useful status checks:

```bash
git status
git log --oneline --decorate --graph --all
```

## 2) Staging and committing

### Stage changes

```bash
git add file.md
git add .
git add -A
```

- `git add file.md`: stage one file.
- `git add .`: stage changes under current directory.
- `git add -A`: stage all changes (new, modified, deleted).

### Why `-a` can be confusing

- `git add -A` (capital `A`) stages all changes.
- `git commit -a` (lowercase `a`) auto-stages only tracked modified/deleted files, not new files.

### Commit staged changes

```bash
git commit -m "Add age summary statistics"
```

## 3) Viewing history and diffs

### History by date range

```bash
git log --since='Apr 2 2024' --until='Apr 11 2024'
```

### Diff basics

```bash
git diff
git diff HEAD
git diff --staged
git diff --staged report.md
git diff main feature-branch
```

- `git diff`: unstaged changes vs index.
- `git diff HEAD`: all local changes vs latest commit.
- `git diff --staged`: staged changes vs latest commit.
- `git diff main feature-branch`: compare two branches.

### Diff hunk header meaning

Example:

```diff
@@ -3,15 +3,33 @@
```

- `-3,15`: old file starts at line 3, spans 15 lines.
- `+3,33`: new file starts at line 3, spans 33 lines.
- Deletions are lines prefixed with `-`; additions are prefixed with `+`.

## 4) Undoing changes before and after commit

### Before commit (`git add` already used)

Unstage one file:

```bash
git restore --staged summary_statistics.csv
```

Unstage all files:

```bash
git restore --staged .
```

Discard local edits in a file (back to `HEAD`):

```bash
git restore summary_statistics.csv
```

Make a file match remote `origin/main`:

```bash
git fetch origin
git restore --source origin/main --staged --worktree summary_statistics.csv
```

### After commit

- Not pushed yet: `git reset --hard HEAD~1` rewrites local history (destructive).
- Already pushed/shared: prefer `git revert <commit>` (safe).

### `git revert` in one sentence

`git revert <commit>` creates a **new commit** that undoes the changes introduced by that commit.

Useful variants:

```bash
git revert HEAD
git revert HEAD --no-edit
git revert HEAD -n
```

### `git checkout HEAD~1 -- report.md` explained simply

Think of it as: “replace my current `report.md` with the version from one commit ago.”

- `HEAD~1` means the previous commit.
- Only `report.md` is restored.
- Then commit if you want to keep that rollback.

Modern equivalent:

```bash
git restore --source=HEAD~1 --staged --worktree report.md
```

## 6) Remotes: clone, fetch, pull, push

### Set up and inspect remotes

```bash
git clone https://github.com/datacamp/project
git remote
git remote -v
git remote add george https://github.com/george_datacamp/repo
```

### Sync commands

```bash
git fetch origin
git pull origin main
git push origin main
```

- `fetch`: download remote updates; no merge.
- `pull`: fetch + merge.
- `push`: publish your local commits.

If push is rejected (non-fast-forward), pull first, resolve any conflicts, then push again.

## 7) Intermediate tools (quick reference)

### Rebase

```bash
git switch feature_branch
git rebase main
git rebase -i <commit-hash>
```

### Cherry-pick

```bash
git cherry-pick <commit-hash>
git cherry-pick <hash1> <hash2>
git cherry-pick --continue
git cherry-pick --abort
```

### Bisect

```bash
git bisect start
git bisect bad
git bisect good <commit-hash>
git bisect reset
```

### Reflog and recovery

```bash
git reflog
git reflog --since="1 week ago"
git checkout <hash>
git checkout -b recovered-branch
```

### Reset modes

```bash
git reset --soft <commit>
git reset --mixed <commit>
git reset --hard <commit>
```

### Worktree

```bash
git worktree add ../etl-bugfix bugfix/data-validation
git worktree list
git worktree remove ../etl-bugfix
```

### Submodule

```bash
git submodule add https://github.com/example/data-validator.git libs/validator
git submodule update --init --remote
git submodule status
```

### Git LFS

```bash
git lfs install
git lfs track "*.csv"
git add .gitattributes
git commit -m "Track CSV files"
```

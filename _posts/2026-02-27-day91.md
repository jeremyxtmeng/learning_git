## Git/GitHub Key Functionalities (Based on My Current Understanding)

- Git helps me track code history, compare changes, and prepare clean commits.
- GitHub is where I store and collaborate on Git repositories online (sharing, pull requests, code review, backup).

### 1) Viewing Commit History
- Command: `git log --since='Apr 2 2024' --until='Apr 11 2024'`
- Meaning: show commits created between those two dates.

### 2) Comparing Changes
- Command: `git diff`
- Correct meaning: shows unstaged changes (working directory vs staging area), not strictly "difference from the last commit."
- If I want to compare against the last commit, I can use: `git diff HEAD`

### 3) Understanding Diff Header
- Example: `@@ -3,15 +3,33 @@`
- Correct meaning:
  - `-3,15` = in the old version, this hunk starts at line 3 and covers 15 lines
  - `+3,33` = in the new version, this hunk starts at line 3 and covers 33 lines
- It does **not** mean "line 3 is deleted."
- Actual deletions are lines prefixed with `-`, additions are lines prefixed with `+`.

### 4) Staging and Reviewing Staged Changes
- Command: `git add file.md`
- Meaning: place the current version of `file.md` into the staging area for the next commit.
- Command: `git diff --staged file.md`
- Meaning: compare staged content of `file.md` against the last commit (what will be committed).

### Why `-a` Sometimes Works in Git

Short answer: for `git add`, the common flag is `-A` (capital A), not `-a`.

- `git add -A` = stage **all** changes (new, modified, deleted).
- `git add .` = stage changes under the current directory.
- `git commit -a` (lowercase `-a`) is different: it auto-stages **tracked** modified/deleted files, but **not** new untracked files.

`git add` / `git commit` change your local repo only. Until `git push origin main`, GitHub online does not change.

## Before commit (you already ran `git add file.md`)

- Unstage only: `git restore --staged file.md`
- Discard local edits and go back to your current local commit (`HEAD`): `git restore file.md`
- Make local file exactly match GitHub `main`:
  1. `git fetch origin`
  2. `git restore --source origin/main --staged --worktree file.md`

## After commit

- If commit is **not pushed**: you can rewrite local history, e.g. `git reset --hard HEAD~1` (destructive).
- If commit is **already pushed** (or shared): use `git revert <commit>` (safe; creates a new commit that undoes it).

## What `git checkout HEAD~1 -- report.md` means

- Think of it as: "replace my current `report.md` with the older version from 1 commit ago."
- `HEAD~1` means "the commit right before my current commit."
- Only `report.md` is changed (not all files).
- After this, commit if you want to keep that rollback.
- Same command in modern style: `git restore --source=HEAD~1 --staged --worktree report.md`.

## What `git revert` means

- “Undo by adding a new commit,” not by deleting history.
    - `git revert <commit>` undoes the changes introduced by that specific commit (which may include multiple files).
    - Git then creates a new commit that records this undo.
    - Original commit stays in history; it is not deleted.

## Case: `git add` used, but not `git commit` yet

You are in the **staging** phase. Nothing is in commit history yet.

### Unstage files
- Unstage a single file: `git restore --staged summary_statistics.csv`
- Unstage all files: `git restore --staged .`

### Typical workflow
1. Edit the file: `summary_statistics.csv`
2. Stage it: `git add summary_statistics.csv`
3. (Optional) Unstage one file: `git restore --staged summary_statistics.csv`
4. (Optional) Unstage all files: `git restore --staged .`
5. Stage again after more edits: `git add summary_statistics.csv`
6. Commit: `git commit -m "Adding age summary statistics"`

### Key point
- `git restore --staged` only affects the staging area.
- Your file content is still there unless you also run a discard command like `git restore summary_statistics.csv`.


## Git Undo/Staging Summary

| Command | Result |
|---|---|
| `git revert HEAD` | Revert the latest commit by creating a **new commit** that undoes it (all files changed in that commit). |
| `git revert HEAD --no-edit` | Same as above, but skip opening the commit message editor. |
| `git revert HEAD -n` | Apply the revert changes to your working tree/index **without creating a commit yet**. |
| `git checkout HEAD~1 -- report.md` | Restore only `report.md` from the previous commit into your working tree (and stage it). |
| `git restore --staged report.md` | Unstage one file (remove it from staging area, keep local edits). |
| `git restore --staged .` | Unstage all staged files. |

### Notes
- `git revert` is safe for shared history because it does not delete old commits.
- `git restore --staged` needs a path (for all files, use `.`).


# Intermediate Git — Merge Conflicts & Remotes (Slide Summary)

Source slide deck: “Merge conflicts” / “Introduction to remotes” (Intermediate Git, George Boorman, DataCamp).



## 1) Merge conflicts: what they are and why they happen

A **merge conflict** occurs when Git cannot automatically reconcile differences between branches—most commonly when **the same file (or the same lines in a file)** are edited on two branches and you try to merge them. Git doesn’t know which version to keep, so it stops and asks you to decide.

Example shown: `README.md` differs between `main` and a `documentation` branch, so `git merge documentation` from `main` triggers a conflict.



## 2) What you see during a conflict (conflict markers)

When you open the conflicted file, Git inserts **conflict markers** to show both versions:

- `<<<<<<< <branch-name>` … content from the other branch (e.g., `documentation`)
- `=======` … separator between versions
- `>>>>>>> HEAD` … content from your current branch (`HEAD`, e.g., `main`)

Your job is to edit the file so it contains the **final desired content**, **removing all markers**.



## 3) How to resolve a merge conflict (typical workflow)

1. **Attempt the merge** (from destination branch):
   - `git merge <source-branch>`
2. Git reports a conflict (e.g., `CONFLICT ... in README.md`)
3. **Open and edit** the conflicted file (example editor: `nano README.md`)
4. **Resolve** by choosing one version, combining both, or rewriting—**and delete the markers**
5. **Stage the resolved file**:
   - `git add README.md`
6. **Commit the resolution**:
   - `git commit -m "Resolving README.md conflict"`
7. After committing, the merge is complete; re-running the merge will show you’re up to date.

**Practical editor tip (nano):**
- Save: `Ctrl + O`, then `Enter`
- Exit: `Ctrl + X`



## 4) Remotes: what they are and why they matter

A **remote repository** is a repo stored elsewhere (often on a hosting service). Typical reasons to use remotes:

- **Backup**: your work exists off your machine.
- **Collaboration**: multiple people can work from different locations.

The deck contrasts:
- **Local repo** on your computer
- **Remote repo** (cloud/hosted)



## 5) Cloning: creating a local copy

Cloning creates a local working copy (sometimes called a “local remote” in the slides):

- Clone a local path:
  - `git clone /path/to/repo`
- Clone a remote URL (hosting services like GitHub/Bitbucket/GitLab):
  - `git clone <URL>`



## 6) Managing remotes: listing, inspecting, adding

After cloning, Git remembers the original location as a remote.

- List remote names:
  - `git remote`
- Show remote URLs (fetch/pull endpoints):
  - `git remote -v`
- Add another remote (useful for collaboration/fork workflows):
  - `git remote add <name> <URL>`



## 7) Fetch vs. merge vs. pull (synchronizing from remotes)

- **Fetch**: downloads remote updates **without** merging into your current branch:
  - `git fetch origin`
  - `git fetch origin main` (fetch only one branch)
- **Merge remote-tracking branch** into your current branch:
  - `git merge origin` (as presented in the deck; conceptually “merge origin’s default branch into current”)
- **Pull** = **fetch + merge** (common workflow):
  - `git pull origin` (default branch into current branch)
  - `git pull origin dev` (fetch `dev` then merge into your *current* local branch)

**Caution:** `git pull` will abort if it would overwrite uncommitted local changes. Save your work first (commit or stash).



## 8) Pushing: sending your local commits to a remote

Pushing shares your local branch history with the remote.

- General form:
  - `git push <remote> <local-branch>`
- Example:
  - `git push origin main`

### Non-fast-forward / “rejected” pushes (remote/local conflicts)

If your local branch is **behind** the remote (remote has commits you don’t), pushing can be rejected with a “non-fast-forward” message.

Typical fix shown:
1. **Pull first** (integrate remote changes):
   - `git pull origin main`
2. Then push again:
   - `git push origin main`

There’s also a “pull without editing” option shown:
- `git pull --no-edit origin main`  
  (the deck notes this is *not recommended* unless you’re very confident in the project history)


## 9) Pushing a new branch to the remote

If you create a new local branch (e.g., `hotfix`) that doesn’t exist on the remote yet:

- `git push origin hotfix`

GitHub may then suggest creating a Pull Request for the new branch.

## 10) Cheat sheet (commands emphasized in the slides)

- Resolve conflicts:
  - `git merge <branch>`
  - edit files → `git add <file>` → `git commit -m "..."`
- Remotes:
  - `git clone <URL>`
  - `git remote`, `git remote -v`
  - `git remote add <name> <URL>`
- Sync:
  - `git fetch origin`
  - `git pull origin`
  - `git push origin <branch>`

# Advanced Git — Merge Types & Advanced Branch Integration (Slide Summary)

> This summary covers the *Advanced Git* slide deck content across: merge strategies, complex merge scenarios, rebasing, cherry-picking, bisecting, history rewriting, recovery tools, and multi-worktree / dependency patterns.

---

## 1) `git merge` and why merge strategy matters

`git merge` combines changes from one branch into another by finding the **common base** between the branches, then applying a merge strategy to integrate the histories. The deck focuses on *how different strategies affect history, collaboration, and debugging*.

---

## 2) Fast-forward merge

### What it is  
A **fast-forward** happens when the target branch has *no new commits* since the source branch diverged. Git can simply move the branch pointer forward—no merge commit is created.

### Why you’d use it  
- Keeps history **simple and linear**
- Great for **short-lived branches** with small, straightforward changes

### When *not* to use it  
- You need to **preserve branch context/history**
- Long-lived feature work
- Conflicts: fast-forward fails immediately if histories can’t be advanced cleanly

### Commands (as shown)
- Default behavior (fast-forward when possible):
  - `git checkout main`
  - `git merge feature-branch`
- Force “only fast-forward”:
  - `git merge --ff-only <branch>`

---

## 3) Recursive merge (`--no-ff`) with a merge commit

### What it is  
A **recursive merge** (typical “true merge”) creates a **merge commit with two parents**, preserving the branching structure.

### Why you’d use it  
- Preserves **historical context**
- Better for **complex or long-lived** development branches
- Makes it easier to see “this feature was merged here” in history

### When *not* to use it  
- You want a very clean, linear history for minor changes

### Command (as shown)
- `git merge --no-ff <branch>`

---

## 4) Squash merge (`--squash`)

### What it does  
A **squash merge** combines *all changes* from a source branch into **one new commit** on the target branch:
- Creates a **regular commit with one parent** (not a multi-parent merge commit)
- Produces a **clean, linear history**
- Does **not** preserve the source branch’s detailed commit history

### Why it’s useful  
- Simplifies code review for large features
- Easier to revert an entire feature as one commit

### Process (as shown)
1. `git checkout main`
2. `git merge --squash data-cleanup`
3. `git commit -m "Implement and optimize data cleanup"`

---

## 5) Octopus merge (`-s octopus`)

### What it does  
An **octopus merge** merges **3+ branches at once** into a single merge commit with **multiple parents**.

### Best use cases  
- Integrating multiple **independent** features simultaneously
- Synchronizing multiple release branches (different versions)
- Works best when branches are unlikely to conflict

### Command (as shown)
- `git merge -s octopus <branch1> <branch2> <branch3> ...`

---

## 6) Rebase (advanced branch integration)

### What rebase is  
`git rebase` integrates changes by **replaying commits** from one branch on top of another—removing merge commits and keeping a **linear commit graph**.

### Key implications  
- Rebased commits get **new hashes**
- Conflicts may need manual resolution during replay
- (Common team rule of thumb): avoid rebasing *public/shared* branches unless you coordinate

### Command (as shown)
- `git rebase <branch-name>`  
Example flow:
1. `git checkout data-cleanup`
2. `git rebase main` (recreates feature commits after `main`’s latest commit)

### Interactive rebase (mentioned)
- `git rebase -i <commit-hash>` to rewrite a range of commits granularly (edit/squash/reorder/etc.)

---

## 7) Cherry-pick (selective commit transfer)

### What it is  
`git cherry-pick` applies the change introduced by **specific commit(s)** onto another branch.

### Typical uses  
- Apply a hotfix across multiple branches
- Bring over a small isolated feature
- Recover a lost commit

### Commands (as shown)
- Single commit: `git cherry-pick <commit-hash>`
- Multiple commits: `git cherry-pick <hash1> <hash2> ...`

### Conflicts & control flags
- Resolve conflicts → `git add <resolved-files>` → `git cherry-pick --continue`
- Abort the cherry-pick: `git cherry-pick --abort`

### Cautions  
Overuse can create duplicate commits and complicate history; consider merge/rebase for larger changes.

---

## 8) `git bisect` (binary search for the “first bad commit”)

### What it is  
A debugging tool that uses **binary search** to find the commit that introduced a regression.

### Workflow (as shown)
1. Start: `git bisect start`
2. Mark current state bad: `git bisect bad`
3. Mark last known good: `git bisect good <commit-hash>`
4. Then repeatedly mark each tested commit:
   - `git bisect good` or `git bisect bad`
5. When done, Git reports the first bad commit; exit:
   - `git bisect reset`

### Automation
- `git bisect run <script-name>`
  - Script returns `0` if tests pass, non-zero if they fail

---

## 9) `git filter-repo` (rewrite history safely/quickly)

### What it is  
A tool to rewrite repository history across **all branches**, often used to:
- Remove sensitive files (tokens/passwords)
- Remove big/unnecessary files
- Restructure repos and reduce size

### Example shown: remove `secrets.txt` from every commit
- Install: `pip install git-filter-repo`
- Rewrite: `git filter-repo --path secrets.txt --invert-paths`

### Implications (very important)
- **All commit hashes change**
- You typically need a **force push**
- Teammates often must **re-clone** (or carefully reconcile)

Best practice: back up and coordinate with collaborators.

---

## 10) `git reflog` (local safety net)

### What it is  
A local record of **all reference updates** (branch tips and `HEAD` moves), stored under `.git/logs/...`.

### Why it matters  
- Helps recover lost commits/branches after resets, rebases, deletes, etc.
- Unlike `git log` (commit history), reflog tracks **actions** and is often temporary (~90 days)

### Useful commands (as shown)
- View reflog: `git reflog` or `git reflog show`
- Cleanup: `git reflog expire`
- Filter by time:
  - `git reflog --since="1 week ago"`
  - `git reflog --until="yesterday"`

### Recover a deleted branch (recipe)
1. Find the commit hash from reflog
2. `git checkout <hash>`
3. `git checkout -b <branch-name>`

---

## 11) `git reset` (move `HEAD`, adjust staging/working tree)

Reset moves `HEAD` to a commit; the *type* determines how index and working directory change:

- **Soft**: `git reset --soft <commit>`
  - working tree unchanged; changes stay **staged**
- **Mixed (default)**: `git reset --mixed <commit>`
  - working tree unchanged; changes become **unstaged**
- **Hard**: `git reset --hard <commit>`
  - discards changes in working tree and staging area (dangerous)

---

## 12) Git worktrees (multiple branches checked out at once)

### What it is  
`git worktree` lets you have **multiple working directories** linked to the same repository, each on a different branch—no constant switching, no stashing.

### Key commands (as shown)
- Add: `git worktree add <path> <branch>`
- List: `git worktree list`
- Remove: `git worktree remove <path>`

### When it shines
- Parallel feature work + urgent hotfixes
- Running tests on multiple branches concurrently
- Reviewing code without disrupting your current work

---

## 13) Submodules (repos inside repos)

### What they are  
A **submodule** is a separate Git repo nested inside another. The main repo points to a *specific commit* of the submodule.

### Key commands (as shown)
- Add: `git submodule add <repo-link|dir> <path>`
- Status: `git submodule status`
- Update:
  - Local init: `git submodule update --init`
  - Remote update: `git submodule update --init --remote`
- Remove (process):
  - `git submodule deinit <name>`
  - `git rm <path>`

Use cases: shared libraries, pinned dependencies, code reuse across projects.

---

## 14) Git LFS (Large File Storage)

### What it is  
`git lfs` stores large/binary files outside the normal Git object store, replacing them with **small pointer files** in the repo.

### Setup flow (as shown)
1. `git lfs install`
2. `git lfs track "*.csv"` (creates/updates `.gitattributes`)
3. `git add .gitattributes`
4. `git commit -m "Track CSV files"`

### Typical workflow
- `git add large_file.csv`
- `git commit -m "..."`
- `git push origin main`
- Pull large content when needed: `git lfs pull`

Best for frequently updated datasets, models, and binaries.

---

## 15) Trunk-based development (TBD)

### What it is  
A branching + CI/CD model where developers integrate into **main** frequently, using **short-lived branches**.

### Core principles (as shown)
- Frequent commits to `main`
- Feature branches are very short-lived (< 1 day)
- Continuous integration
- Feature flags for incomplete work

### Benefits vs challenges
- Benefits: fewer merge conflicts, faster releases, improved quality/collaboration
- Challenges: requires discipline, robust testing, feature flag management

## Command cheat sheet (by concept)

- **Fast-forward merge**: `git merge --ff-only <branch>`
- **Merge commit**: `git merge --no-ff <branch>`
- **Squash merge**: `git merge --squash <branch>` → `git commit -m "..."`
- **Octopus merge**: `git merge -s octopus b1 b2 b3`
- **Rebase**: `git rebase main`, `git rebase -i <hash>`
- **Cherry-pick**: `git cherry-pick <hash>` (+ `--continue` / `--abort`)
- **Bisect**: `git bisect start` → `bad/good` loop → `git bisect reset`
- **History rewrite**: `git filter-repo ...` (coordinate + expect force push)
- **Recovery**: `git reflog` (then checkout hash)
- **Parallel branches**: `git worktree add/list/remove`
- **Dependencies**: `git submodule add/update/status`
- **Large files**: `git lfs install/track/pull`
